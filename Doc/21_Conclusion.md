# 21. Заключение и принципы

## Ключевые принципы архитектуры

1. **Dependency Injection** — всё, что можно, регистрируется в контейнере. Зависимости разрешаются через иерархию контейнеров.

2. **Разделение ответственности** — сервисы (бизнес-логика), команды/запросы (изменение/чтение данных), View (отображение).

3. **Реактивность** — изменения данных автоматически обновляют UI через подписки R3.

4. **Иерархия контейнеров** — корневые зависимости (глобальные сервисы) доступны на всех сценах, сценарные — только в рамках сцены.

5. **CQRS** — явное разделение команд (изменение) и запросов (чтение). Команды автоматически сохраняют состояние.

6. **FSM** — каждое состояние игры моделируется конечным автоматом. UI и сервисы подписываются на смену состояний.

7. **Proxy для состояний** — POCO-данные оборачиваются в реактивные Proxy с автосинхронизацией.

8. **Автосохранение** — после каждой успешной команды состояние автоматически сохраняется (локально + на сервер).

## Обобщённый поток данных

```
User Input (клик/кнопка)
    → ViewModel: Subject.OnNext()
    → Service: бизнес-логика, вызов Command
    → CommandHandler: создание/изменение EntityData
    → GameplayStateProxy (ObservableList / ReactiveProperty)
    → Service (подписка): создание ViewModel
    → ViewModel (R3.Subscribe)
    → Binder (обновление Unity-компонентов)
    → Визуальное изменение на экране
```

## Типичный жизненный цикл объекта в игре

1. **Создание EntityData** — команда создаёт POCO-данные с ID, позицией, параметрами
2. **Entity (Proxy)** — данные оборачиваются в реактивную сущность
3. **Добавление в ObservableList** — список уведомляет подписчиков
4. **Service реагирует** — создаёт ViewModel, настраивает её
5. **View связывается** — Binder подписывается на ViewModel
6. **Жизнь объекта** — объект обновляется, взаимодействует
7. **Удаление** — команда удаляет EntityData → ObservableList уведомляет → Service дизпозит ViewModel → Binder удаляется
8. **Очистка** — при выходе со сцены все сервисы дизпозятся, удаляя все ViewModel и Binder

## Что стоит улучшить / заметки

- **Дублирование состояния** — из-за паттерна Proxy часть данных существует в двух копиях (POCO + Proxy). Следить за синхронизацией.
- **Сложность отладки** — реактивные цепочки могут быть сложны для отладки. Рекомендуется логирование ключевых подписок.
- **Порядок регистрации** — важен порядок создания сервисов из-за зависимостей между ними.
- **Память** — при большом количестве подписок нужно следить за своевременным Dispose.
